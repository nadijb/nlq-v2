### 3. Database Schema

Make sure your Supabase `messages` table has the following schema:

```sql
create table public.messages (
  id bigint generated by default as identity not null,
  author character varying not null, -- 'USER' or 'AI'
  content text not null,
  created_at timestamp with time zone not null default now(),
  session_id character varying not null,
  constraint messages_pkey primary key (id),
  constraint messages_id_key unique (id)
) TABLESPACE pg_default;
```

## Architecture

### Backend Integration

- **Supabase**: Used for retrieving messages and conversations (read-only from frontend)
- **n8n**: Handles AI processing and message insertion into Supabase
- **API Routes**: Server-side Next.js API routes handle all backend communication to avoid CORS issues

### Session Management

- Each conversation uses a `session_id` in the format: `chat_{timestamp}`
- Session IDs are generated client-side when creating a new conversation
- Messages are grouped by `session_id` in Supabase

### API Endpoints

- `GET /api/conversations` - Fetch all conversations
- `GET /api/messages/[sessionId]` - Fetch messages for a specific session
- `POST /api/chat` - Send message to n8n webhook

## Usage

1. **Create a new conversation**: Click "New Conversation" button in the sidebar
2. **Send a message**: Type your message and press Enter or click Send
3. **View history**: Click on any conversation in the sidebar to view its messages
4. **Delete conversation**: Hover over a conversation and click the delete icon
5. **Copy messages**: Hover over any message and click the copy icon

## Notes

- Message insertion is handled by your n8n workflow, not directly from the frontend
- The frontend only reads from Supabase and sends messages to n8n
- All backend communication happens server-side to avoid CORS issues
